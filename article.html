Por qué este proyecto es distinto

No es un concepto. Es un producto real donde Claude Code se conectó directo a Figma vía MCP y el diseño ocurrió desde la terminal, con criterio humano en cada decisión.

"Mientras muchos corrían asustados por lo que la IA podría quitarles, yo elegí aprender qué podía darme."

El resultado: Isla Nativa Travel, un marketplace de experiencias turísticas en Florianópolis, Brasil, construido de inicio a fin con este flujo.

Qué construí: resumen ejecutivo
  • Producto: Marketplace digital para booking de experiencias turísticas (Florianópolis, Brasil)
  • Alcance: End-to-end: discovery, arquitectura, UI, design system y handoff a desarrollo
  • Pantallas: Home completa en 3 breakpoints (desktop 1920px · tablet 768px · mobile 390px)
  • Design System: 193 variables · 5 colecciones · 14 conjuntos de componentes · 16 estilos de texto · 0 valores hardcoded
  • Handoff: Tokens exportados a CSS · componentes traducidos a Vue.js con misma nomenclatura
  • Herramienta diferencial: Claude Code + MCP de Figma: terminal conectada a Figma Desktop en tiempo real, sin abrir el editor

Las cinco fases: cómo lo hice

Isla Nativa conecta viajeros con guías locales en Florianópolis. El proyecto fue end-to-end: discovery, design system, UI en 3 breakpoints y handoff completo a Vue.js.

1
Discovery e Ideación

Del briefing a los insights en horas

Analicé el mercado, benchmarkeé competidores (Airbnb Experiences, Viator, GetYourGuide) y sinteticé oportunidades en días, no semanas. Pero la IA no tomó las decisiones estratégicas. Yo sí. Ella organizó; yo determiné qué importaba.

2
Arquitectura de Producto

Estructura que se valida antes de diseñar

Definimos juntos 7 módulos del producto: Discovery Engine, Experience Catalog, Booking System, Pricing Transparency, Trust & Reviews, Comparison Engine y User Profiles. La IA cuestionó cada decisión y documentó la arquitectura al nivel de detalle que normalmente no da el tiempo.

3
Diseño UI

Layouts generados, refinados por criterio

Claude Code con MCP de Figma generaba los frames directamente en el canvas. En lugar de dibujar pixel a pixel, describía la estructura y el resultado aparecía en Figma en segundos. El tiempo fue para las decisiones, no para la ejecución.

4
Design System

193 variables. Cero hardcode. Documentación completa.

El Design System se construyó en paralelo con el diseño, no al final. Cada decisión visual se convirtió en un token. Cada componente se documentó en el momento de su creación. El resultado: un sistema listo para escalar desde el primer día.

5
Handoff y Desarrollo

Del diseño a Vue.js sin fricción

Los tokens se exportaron directamente a CSS. Los componentes Figma se tradujeron a Vue.js con la misma nomenclatura y estructura. La coherencia entre diseño y código no fue un objetivo. Fue una consecuencia del proceso.

Claude Code + MCP Figma: el stack que lo hizo posible

MCP (Model Context Protocol) es el protocolo que permite a Claude Code hablar directamente con Figma Desktop. Sin copiar y pegar. Sin cambiar de ventana. Un comando en la terminal, el resultado en el canvas al instante.

claude-code · figma-mcp 
$ node src/index.js render '<Frame bg="var:card" stroke="var:border" rounded={12} p={24} flex="col" gap={16}>
  <Text color="var:foreground" size={18} weight="bold" w="fill">Experiencias destacadas</Text>
  <ExperienceCard />
</Frame>'

✓ Frame created  · ID 847:23
✓ Variables bound · card, foreground, border
✓ Reflected in Figma Desktop
Lo que habilitó este stack
  • ⚙️ Frames y componentes desde terminal: secciones completas generadas con código JSX, reflejadas en Figma en tiempo real
  • 🎨 Variables bindeadas al crear: var:card, var:border, var:primary. Cero hardcode en ningún componente
  • 📤 Tokens exportados a CSS: los mismos nombres en Figma y en el código Vue. Sin traducción manual
  • 🧩 Misma nomenclatura en todo el sistema: TheNavBar, ExperienceCard, AppButton en Figma y en Vue.js son el mismo componente

Un desarrollador que recibe este handoff no necesita interpretar el diseño. Todo lo que ve en Figma tiene correspondencia directa en el código. Eso elimina la fricción más costosa del proceso.

El Design System: rigor desde el inicio

El DS no fue un entregable final. Fue la columna vertebral del proyecto desde la primera decisión visual.

  • 193 variables · 0 valores hardcoded
  • 5 colecciones: Color · Spacing · Typography · Radius · Components
  • 14 conjuntos de componentes con variantes y estados completos
  • Soporte Light / Dark mode · Documentación inline

Cada token tiene un nombre semántico: color/text/brand, spacing/16, radius/xl. Cuando cambia un token, el sistema entero se actualiza.

Lo que cambió en mi forma de diseñar

"El diseñador del futuro no será el que dibuja más rápido. Será el que toma mejores decisiones con más información, en menos tiempo."

Claude Code con MCP de Figma no me quitó trabajo. Me quitó la parte mecánica: replicar variantes, mantener consistencia, exportar especificaciones. Lo que quedó fue espacio para lo que realmente define el buen diseño: criterio, empatía, coherencia de experiencia.
